# Advanced Portfolio PRD + Wireframe

## Product Name
Shriram M — International-Style Engineering Portfolio

## Objective
Create a premium developer portfolio that positions Shriram as a serious early-career **Backend / Full Stack / AI Engineer** instead of a generic fresher portfolio.

The site should communicate:
- strong backend engineering fundamentals,
- product-building ability,
- real AI / RAG / Spring AI exposure,
- internship-based practical experience,
- readiness for startup, SaaS, and product engineering roles.

---

## Primary Goal
Convert visitors into one of these actions:
- View featured projects
- Download resume
- Open GitHub / LinkedIn
- Contact for interview or hiring discussion

---

## Target Audience

### Primary Audience
- Recruiters hiring for Backend Engineer, Full Stack Engineer, AI Backend Engineer, Software Engineer I, Product Engineer
- Startup founders hiring early engineers
- Engineering managers evaluating practical fit

### Secondary Audience
- Developers reviewing technical credibility
- Hiring teams screening product thinking and execution ability

---

## Core Positioning
The portfolio should make the visitor think:

> This candidate is not just a student. He builds real backend systems, ships products, understands AI workflows, and can contribute to production engineering teams.

### Positioning Statement
> Backend & AI Engineer building full-stack products, backend systems, and AI-powered workflows with Java, Spring Boot, React, Next.js, and LLM tooling.

---

## Design Direction

### Visual Style
- Minimal
- Premium
- International startup aesthetic
- Engineering-first
- Product-focused
- Calm and high-signal

### Design Keywords
- sharp
- clean
- credible
- modern
- focused
- technical

### Avoid
- generic portfolio template look
- overly decorative 3D hero visuals
- auto-generated GitHub repo sections
- random gradients and flashy animations
- buzzword-heavy copy
- large empty areas without purpose

---

## Information Architecture

### Final Section Order
1. Header / Navigation
2. Hero
3. Credibility Strip
4. About
5. Experience
6. Featured Projects
7. Skills
8. Optional “Currently Exploring / Building”
9. Contact CTA
10. Footer

---

## Page-Level UX Principles
- The first screen should immediately show role, stack, and value
- Every section should answer “why hire this person?”
- Projects should feel like case studies, not random cards
- GitHub should support credibility, not become the main content
- The site should scan well within 10–20 seconds
- Mobile version must preserve clarity and hierarchy

---

- Built and shipped FeedbackWall

- Focused on backend, AI backend, and full-stack roles

Horizontal row of compact cards or proof badges.

Do **not** show raw GitHub follower counts or repository counts here.

## 4. About Section

### Purpose
Explain who Shriram is, what he builds, and what kind of role he is seeking.

### Layout
Two-column layout:
- Left: short paragraph-based intro
- Right: quick facts card

### Recommended Copy
**Section Title**  
About

**Body**  
Shriram is a software engineer focused on backend systems, full-stack product development, and applied AI. His work centers on turning complex ideas into practical software — especially where APIs, automation, data flow, and LLM-powered features come together. He is currently looking for entry-level roles where he can contribute to backend engineering, AI applications, or product-focused full-stack teams.

### Quick Facts Card
- Based in Tamil Nadu, India
- 2026 Computer Science graduate
- Interested in Backend / AI Backend / Full Stack roles
- Core stack: Java, Spring Boot, React, Next.js, Python
- Open to full-time opportunities

### Writing Rules
- Keep paragraphs short
- Remove generic aspirational lines
- Focus on capability + direction

## 5. Experience Section

### Purpose
Show real-world engineering exposure and improve recruiter confidence.

### Layout
Timeline or stacked cards.

### Structure per Role
- Company name
- Role
- Duration
- 3–4 bullets

### Bullet Style Rule
Every bullet should show:
- what was built,
- which technologies were used,
- and why it mattered.

### Recommended Entries

#### Servimos Technologies / WeServeTech LLC — Software Development Engineer Intern
- Implemented MCP-style server functionality using SSE transport, JSON-RPC, and tool-calling patterns.
- Integrated Spring AI into Java-based applications to connect backend services with LLM-powered workflows.
- Built and tested RAG pipeline components for retrieval-based question answering and automation use cases.
- Contributed to documentation and real-world Agile engineering workflows.

#### Edunet — Intern
- Add final verified contributions.
- Keep to 2–3 strong bullets only.

### Avoid
- “learned a lot”
- “gained exposure”
- “worked in a team” without evidence of actual work

---

## 6. Featured Projects

### Purpose
This should become the strongest section on the entire site.

### Rule
Only feature the top 3 projects.

### Recommended Projects
1. FeedbackWall
2. AI / RAG / Spring AI project
3. Multigate EduFlow

### Card Structure
Each project must include:
- Project name
- Short problem statement
- Short solution summary
- Stack tags
- Key technical highlight
- Buttons:
  - Live Demo
  - GitHub
  - Case Study (optional)

### Project Copy Framework
**Project Name**  
One-line description of the problem it solves.

**What was built**  
A practical explanation of the product or system.

**Tech Stack**  
List of key technologies.

**Why it matters**  
What makes this technically or product-wise impressive.

### Example Format
**FeedbackWall**  
A SaaS product for collecting, organizing, and prioritizing user feedback and feature requests.  
Built with Next.js and Supabase, focused on practical product workflow and clean MVP execution.  
Highlights: authentication, structured feedback pipeline, prioritization workflow, and SaaS-style UX.

### Design Direction
Project cards should feel like mini case studies.

### Required Enhancements
- Larger cards
- Better hierarchy
- Use screenshots or interface previews where possible
- Keep only strong projects
- Remove weak / empty GitHub repos from main showcase

- CTA buttons

### Avoid

- noisy shadows
- neon accents
- unnecessary animation

### Keep
- subtle hover effects
- smooth anchor scroll

- polished button interactions

### Avoid
- dramatic entrance animations
- overuse of motion
- distracting counters or gimmicks

## Success Criteria
The new portfolio succeeds if:
- recruiters understand the candidate’s role fit in under 10 seconds,
- projects feel curated and credible,
- experience reads like real engineering work,
- the site feels closer to global product-engineer portfolios than student templates,
- visitors are more likely to open resume, GitHub, and contact links.
